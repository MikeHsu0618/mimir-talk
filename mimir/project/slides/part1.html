§ 2 · Architecture Credit where credit is due 08
SIDECAR 的巧妙之處

它不重工，效率至上。

關鍵洞察

S3 上的 block 跟本地 disk 完全一樣。
Sidecar 不做任何運算。

對比 Remote-Write

Backend 要解 protobuf、重壓縮、重建 index——
同一份運算做了兩次。

所以當初選 Sidecar，是合理的。

§ 3 · Pain & Selection Store Gateway, up close 11
痛點 ② 長期查詢

Thanos Store Gateway
撐不住我們這個量級。

① 啟動慢

冷啟動數十分鐘。

啟動要先 sync bucket + 下載 index header， block 越多越慢——擴 Pod、rolling update、節點重建都在等它。

② Sharding 不靈活

靜態切 · relabel 硬刻。

不像 Mimir 的 consistent hash ring—— Thanos 得用 relabel 規則把 block 綁在特定 SG 上，改一次要重啟。

③ 擴展困難

水平加機器 ≠ 加容量。

新 SG 必須重新 sync 自己負責的 block，熱資料得重刷一遍 cache—— 擴容期間整體反而變慢。

④ Cache 碎片化

每個 SG 各自為政。

Index cache、chunks cache 各自獨立—— 同一份熱 block 在多台 SG 重複佔記憶體，命中率低、OOM 容易。

四件事都指向同一個根——SG 是 stateful 的。 擴 / 縮 / 改 / 重開，都要重新同步。
